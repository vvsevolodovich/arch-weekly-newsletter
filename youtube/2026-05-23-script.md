## MySQL Beats Redis at Shopify Scale

### THE PROBLEM

You've probably hit this if you've ever built inventory reservations at scale. You grab Redis because INCR and DECR are atomic. Then you need the reservation to survive a payment — and now you've got two systems that can drift. Payment succeeds, the Redis claim never lands, you've oversold. Or you block on a single counter and throughput falls off a cliff.

### WHAT THEY DID

Shopify scrapped the counter model entirely. Instead of one quantity-per-item row, they create one row per sellable unit. MySQL 8's SKIP LOCKED lets concurrent transactions each claim a different row without blocking each other. A cap of one thousand rows per item-location pair keeps write volume bounded. Reserve and claim now happen in a single ACID transaction — the two-system drift problem simply doesn't exist.

### MY TAKE

This works because inventory is discrete. You're selling units, not fluid. Row-per-unit maps directly to the domain model. The hidden assumption is that concurrent contention stays well under that thousand-row ceiling — a flash sale for limited-edition items could stress-test it. But for most commerce workloads, this is cleaner and more correct than Redis ever was. The cargo-culting runs deep in our industry: Redis is a cache, not an inventory ledger.

### TAKEAWAY

If you're building any reservation system today — inventory, seats, hotel rooms, parking spots — model discrete units as discrete rows and reach for SKIP LOCKED. Stop assuming you need a cache layer for throughput.

## AI Agents and the Feedback Loop Problem

### THE PROBLEM

Everyone's asking which tasks AI agents can handle. The usual answer involves benchmarks and model capability. Marc Brooker from AWS thinks that's the wrong axis entirely.

### WHAT THEY DID

Brooker maps task suitability to feedback speed and clarity. Building a UI? Fast, visual, objective feedback — agents do well. Writing concurrent code? Bugs manifest as silent data corruption at runtime — agents struggle because there's no tight feedback signal. Architecture decisions? Feedback is contextual, delayed, and often never arrives at all. The claim: this single axis predicts agent performance better than any model benchmark.

### MY TAKE

This is the most useful framing I've seen for evaluating agent deployment. It also explains something that wasn't obvious before: why agents with tool use are so much better than agents in isolation. Each tool call is a feedback tick. The more tools, the tighter the loop. It also tells you exactly where not to aim agents first — anything where the feedback is slow, silent, or subjective. Which is, unfortunately, most of the hard engineering problems.

### TAKEAWAY

Before you automate any engineering task with agents, ask: what's the feedback signal, and how fast does it arrive? If the answer is "slow and ambiguous," you're setting the agent up to fail silently.

## Cloudflare Workflows V2: 11× Concurrency

### THE PROBLEM

You're running multi-step workflows at scale — AI agent pipelines, long-running background jobs. Your coordinator is a monolithic account-level gatekeeper. As concurrent instances grow, it becomes the bottleneck. You can't scale the executor without also scaling the coordinator, and the coordinator is a single point of contention.

### WHAT THEY DID

Cloudflare split the coordinator into two primitives. SousChef manages lifecycle metadata for a per-account subset of workflow instances. Gatekeeper distributes concurrency slots across all SousChef nodes. Metadata management is now separate from execution. Each workflow step is isolated, replayable, and idempotent — a worker restart resumes from the last committed checkpoint, not from scratch. The result: 4,500 concurrent instances became 50,000. Creation rate tripled to 300 starts per second. Queue depth hit two million instances per workflow.

### MY TAKE

The decomposition is textbook — separate metadata from execution, distribute the bottleneck. What's interesting is this was clearly designed with agentic workloads in mind: long-running, multi-step, durable. The constraint they don't address is replay determinism. If your steps have observable side effects, idempotency is your problem to solve, not theirs. They give you the checkpoint; you have to make sure replaying from it doesn't send the same email twice.

### TAKEAWAY

If you're hitting orchestration ceilings, split metadata coordination from execution slot distribution. Don't let a single coordinator own both concerns.

## Databricks Spark Connect

### THE PROBLEM

You have a multi-tenant Spark cluster. One job goes memory-hungry and OOMs the driver. Adjacent jobs that had nothing to do with it start failing too. The blast radius is the entire cluster because user application code runs inside the driver process, sharing its failure domain.

### WHAT THEY DID

Spark Connect is a gRPC split that moves user code outside the driver entirely. The driver becomes a query coordinator — no user code, no shared failure domain. A Serverless Gateway handles multi-tenant workload routing. An adaptive autoscaler right-sizes capacity continuously for both cost and latency. The result: query-level isolation, and 25-plus major Spark runtime upgrades per year delivered at 99.998% success rate across 4.5 billion workloads — transparently, with zero user action.

### MY TAKE

Moving user code out of the driver is the natural extension of how databases have always separated client from server. It just took the Spark ecosystem a decade to get there. The upgrade success rate is genuinely impressive — though it's a Databricks-managed fleet, so they control the rollout conditions. The important architectural takeaway is simpler: any system where tenants share a driver process shares a failure domain. That's the invariant to break.

### TAKEAWAY

If you're running multi-tenant Spark, ask whether user code runs inside the driver. If it does, you're one OOM away from a cluster-wide incident. The Spark Connect split is the pattern to adopt.

## Uber Agent Identity

### THE PROBLEM

AI agents don't fit the two-category model that most Zero Trust architectures were built around: humans and services. An agent acts on behalf of a human, chains through other agents, invokes tools, and leaves an audit trail — or doesn't. Your existing identity system has no clean answer for what principal an agent represents mid-chain.

### WHAT THEY DID

Uber built three chained primitives. An Agent Registry maps every agent to a verified Michelangelo ML workload, establishing a cryptographic principal. A Security Token Service issues per-hop JWTs scoped to specific destinations with short TTLs, preventing replay. An MCP Gateway enforces these policies at every tool invocation boundary. Each token embeds the full actor chain — from the originating user through every intermediate agent — giving end-to-end traceability. They're aligning the design to IETF WIMSE drafts to track emerging standards.

### MY TAKE

Per-hop JWTs are the right call — they limit blast radius if a token leaks. The thing I'd push back on: short TTLs create a token refresh problem for long-running agent pipelines. If your chain takes ten minutes, you need careful refresh logic or you'll hit expiry mid-invocation. The WIMSE alignment is smart. This problem needs a standard before every company builds incompatible solutions.

### TAKEAWAY

If you're building production agent infrastructure, define your identity model before your first agent goes live. Retrofitting Zero Trust onto agentic systems after the fact is a miserable project.

## JFrog: Supply Chain on Fire

### THE PROBLEM

You've locked down your npm registry, you're scanning for CVEs, you think your supply chain posture is reasonable. JFrog analyzed 18.2 billion artifacts in 2025 and found attack vectors that didn't exist on anyone's radar two years ago.

### WHAT THEY DID

Four new categories emerged at scale: 177,000 new malicious packages including a 451% npm surge. 495 malicious AI models on Hugging Face. 969 malicious AI agent skills carrying high-impact payloads. 56 malicious extensions on OpenVSX. One campaign — "Qix" — used 25 packages to compromise 2.5 million downloads. Despite all of this, 53% of organizations still pull AI models from public registries where malicious payloads have been confirmed. Only 28% have secrets detection active.

### MY TAKE

The new attack surface is the AI model and agent skill registry, and almost no organization treats it with the same scrutiny they'd apply to an npm package. If you're pulling a fine-tuned model from Hugging Face without artifact signing or provenance verification, you're doing the equivalent of running a random npm install with no lockfile and no audit. Your supply chain problem just grew three new dimensions. The tooling hasn't caught up yet.

### TAKEAWAY

Treat AI models and agent skills with the same governance you apply to software dependencies: artifact signing, provenance verification, and private registries for approved models only.

## Temporal for Durable Execution

### THE PROBLEM

You've got a workflow that chains API calls, waits on external systems, might run for hours. You build it with cron plus a queue plus a database to track state. Then your worker crashes mid-chain. Now the recovery logic is as complex as the business logic. Maybe more.

### WHAT THEY DID

Temporal records every workflow event — activity scheduled, timer fired, signal received — as an immutable entry in an event history persisted to Cassandra, Postgres, or MySQL. On worker restart, it replays that history so your process resumes from exactly where it left off, with no custom recovery code. Side effects live in Activities with independent retry policies. Workflow code must be strictly deterministic: no random values, no wall-clock reads, no direct I/O.

### MY TAKE

The determinism constraint is the hidden tax. Any developer who's debugged a non-deterministic replay has lost hours to it — and the error messages are not friendly. But for agentic systems chaining dozens of tool calls over minutes or hours, this durability model solves exactly the failure class that cron-queue-DB breaks under. The alternative doesn't just have more code; it has failure modes that are much harder to reproduce and debug.

### TAKEAWAY

If your workflow has more than two steps and any of them can fail independently, Temporal's event-sourcing model is worth the determinism learning curve.

## API Governance Is Your MCP Readiness

### THE PROBLEM

Your org wants to wire up AI agents via MCP. You're looking at SDK documentation and integration guides. The actual bottleneck is almost certainly not the implementation — it's whether you have machine-readable API definitions and consistent permission boundaries at all.

### WHAT THEY DID

MCP is architecturally simple: a long-lived HTTP connection serving JSON. The constraint is your underlying governance stack: OpenAPI specs, developer portals, access control, rate limiting. Enterprises with documented APIs and fine-grained security policies are discovering those investments were preparation for exactly this transition. Organizations sitting on undocumented REST sprawl have a deeper remediation problem than any SDK can solve.

### MY TAKE

This is right, but it understates one gap: even with solid OpenAPI specs, MCP exposes APIs to agents that invoke them at machine speed without human judgment. Rate limits and access controls designed for human usage patterns will break under agentic load. You don't just need governance; you need governance calibrated for automated callers that don't read error messages and will retry at full throttle.

### TAKEAWAY

Before you implement MCP, audit your API documentation coverage. If you can't describe an API in machine-readable form, you can't safely expose it to an agent.

## Valkey 9.1

### THE PROBLEM

Redis forked into Valkey in 2024 after the license change. A year and a half in, the question is whether Valkey is shipping meaningful improvements or just maintaining compatibility.

### WHAT THEY DID

Valkey 9.1 delivers real changes: up to 20% memory reduction for strings under 128 bytes, up to 10% for sorted sets via skiplist optimizations — no reconfiguration required. A redesigned I/O threading model improves throughput by up to 17%. A new database-level ACL system enables fine-grained multi-tenant isolation within a single instance. And Valkey Search 1.2 folds full-text search, tag filtering, numeric lookups, and vector search directly into the cache — no separate search platform needed.

### MY TAKE

The integrated search story is the move worth watching. Running vector search in the same system as your cache is directly useful for retrieval-augmented generation — you don't need a separate vector database if your cache already handles semantic lookup. The performance gains are real but not dramatic enough to force a migration on their own. The combined memory savings plus integrated search is the actual value proposition.

### TAKEAWAY

If you're evaluating caching infrastructure for AI-adjacent workloads that need semantic search, Valkey 9.1's integrated search layer is worth putting in the evaluation matrix.

## Google I/O Agent Platform

### THE PROBLEM

Every team building AI agents is reinventing orchestration, security, and identity from scratch. No-code platforms can't scale. Code-first frameworks lock out product teams. Agent-to-agent communication has no shared standard, so every integration is a bespoke contract.

### WHAT THEY DID

Google formalized a four-tier stack: Agent Studio for no-code, a Managed Agents API with sandboxed execution per run, Antigravity 2.0 for multi-agent orchestration, and ADK 2.0 for code-first development. All four tiers are unified by the A2A protocol — any agent at any level can be called as a sub-agent by another. ADK 2.0 adds three sub-agent execution modes: chat for full interaction, task for auto-return-to-parent, and single-turn for parallel invocation. A governance layer handles identity, gateway, security, and registry.

### MY TAKE

The A2A protocol is the piece worth tracking. If it gets adoption beyond Google's own ecosystem, it becomes the foundation for interoperable agent infrastructure across vendors. The four-tier abstraction is sensible on paper, but platform-of-everything announcements have a long history of being impressive at launch and abandoned two years later. Adoption signal matters more than architecture diagrams.

### TAKEAWAY

Watch the A2A protocol adoption, not the individual tools. Interoperability between agent systems from different vendors is the problem worth solving at the standards level.

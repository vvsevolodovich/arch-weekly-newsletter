## AGENTJACKING: 85% HIJACK RATE VIA FAKE BUG REPORTS

### THE PROBLEM

You've connected your AI coding agent to your error tracker because triage is tedious and the agent is good at reading stack traces. Every team running Claude Code, Cursor, or Codex through an MCP-connected observability tool has implicitly extended trust to every event that tool ingests. At real volume — hundreds of error events a day — nobody is reading the raw payload before the agent does. You've built a pipeline that auto-feeds unvetted text straight into an agent with write access to your repo.

### WHAT THEY DID

Tenet Security proved that trust is misplaced. With nothing but a public write-only Sentry DSN key and one HTTP POST, they injected attacker-controlled instructions into an error event. The coding agent picked it up through the Sentry MCP server and executed those instructions as legitimate remediation guidance. They call it the "Authorized Intent Chain" — every step is technically authorized, so none of your perimeter defenses ever fire, because nothing looks unauthorized. Success rate: 85% across more than 100 real-world targets. Sentry acknowledged the report on June 3 and declined to fix it at the platform level.

### MY TAKE

Sentry's call here is defensible, even though it sounds like a dodge. A DSN write key is intentionally public and low-friction — that's the point of the product, letting any client report errors without an auth handshake. Sentry can't turn that into content moderation for arbitrary stack trace text without breaking the product for everyone. What's actually broken is the agent's authorization model: it treats "this came from a trusted internal tool" as "this is a trusted instruction," with zero concept of provenance once data crosses a system boundary. That's not Sentry's bug. That's every MCP integration's bug.

### THE TAKEAWAY

If your coding agent pulls from anything with public or low-friction ingestion — error trackers, support tickets, webhooks, customer-submitted forms — treat that content as untrusted input the moment it enters the agent's context, no matter how official it's formatted. The fix has to live in the agent runtime, not upstream.

## GITHUB: THE 32-BIT INTEGER THAT BROKE PR REVIEWS

### THE PROBLEM

You've inherited a sharding scheme where "someone migrated the important table to 64-bit years ago" is a sentence you trust without checking whether it covers every table touching that ID space. At GitHub's scale, that gap doesn't surface for years — until it does, all at once, in production, during business hours.

### WHAT THEY DID

On May 6, a 32-bit integer column in a Vitess lookup table for PR review threads exhausted its address space. Nearly 100% of new review thread creation failed for 3 hours and 50 minutes. The primary table had already been migrated to 64-bit integers; the secondary Vitess lookup table mapping the same IDs had not, and nothing in their alerting tracked the drift between the two. GitHub's public response was to raise its capacity-planning target from 10x headroom to 30x, citing AI coding agent write volume as part of what's accelerating growth.

### MY TAKE

The overflow itself is the boring part — integers running out is inevitable and well understood. The actual failure is that a migration got marked "done" when it covered the primary table but not the derived lookup structure sitting right next to it, and nobody had "distance to int32 max" as a monitored metric on either one. Bumping the headroom target from 10x to 30x is a band-aid, not a fix — it buys time on this specific dimension but does nothing for the next derived table built on the same key space that someone forgets to re-audit during the next migration.

### THE TAKEAWAY

Any time you migrate a primary key's data type, go find every secondary index, cache, and lookup structure built against that same ID space and verify independently that it was migrated too. "We already fixed this" is not an audit.

## AWS: 69% FEWER ROUTERS VIA 1990S MATH

### THE PROBLEM

You've scaled a network on the standard fat-tree topology, where more capacity means more layers, more routers, more cabling, and a combinatorial mess of failure domains. Every hyperscaler eventually hits the point where the hierarchy itself, not the bandwidth, is the bottleneck — and the router count becomes the line item nobody can justify anymore.

### WHAT THEY DID

Amazon's Resilient Network Graphs scrap the fat-tree hierarchy for a flat, quasi-random mesh connecting top-of-rack switches directly to each other through passive optical "ShuffleBoxes" — applying a result from early-1990s expander graph theory showing that random connectivity beats hierarchical structure for resilience and efficiency at scale. The numbers: 69% fewer routers, 33% more throughput, 40% less power. First production rollout was Dublin, late 2024; by April 2026 it's the default for most new AWS data centers globally — the largest known production deployment of expander-graph topology by any cloud provider.

### MY TAKE

The interesting part isn't the math, it's the 30-year gap before anyone shipped it. Fat-tree's predictable, hand-traceable structure was worth the router tax right up until the tax got expensive enough to justify the operational cost of a topology a human can't intuitively reason about — quasi-random routing needs tooling just to visualize, let alone debug when a link goes down. That's the right trade at AWS's scale, where the tooling already exists. It would be a terrible trade for a team running three data centers without that investment, because your on-call engineer needs to picture the network when something breaks at 3am.

### THE TAKEAWAY

Don't adopt expander-graph topology because the math is elegant. Adopt it once fat-tree's router count is your actual cost driver, and only after you've built the tooling to debug a topology you can't trace by hand.

## MULTI-AGENT CODING HITS THE FLP WALL

### THE PROBLEM

You've run multiple AI agents against the same codebase expecting them to coordinate like well-behaved distributed workers, and instead watched them quietly misinterpret each other's intent — no crash, no error, just confidently wrong output that looks completely plausible until someone reviews it.

### WHAT THEY DID

The argument here is that this isn't a "wait for better models" problem — it's the FLP impossibility theorem in a new outfit. FLP says you can't guarantee consensus in bounded time over an asynchronous network where messages can be delayed indefinitely; message delivery to an LLM agent fits that exactly, because the agent decides when, or whether, to act on what it's handed. The piece reframes "the agent misread the instruction" as the LLM-world equivalent of a crash fault, and proposes the same fix distributed systems have always used: external validation — tests, static analysis, formal verification — that converts a silent misinterpretation into a detectable, visible failure.

### MY TAKE

The reframe is sharp, and the theory checks out, but it's doing a bit of rhetorical heavy lifting. FLP is about guaranteeing consensus in the worst case; most multi-agent coding setups don't need a guarantee, they need "good enough most of the time with a human or a test suite as the backstop." Strip away the academic packaging and the actionable insight is simpler than the theorem: don't build a pipeline that assumes agreement, build one where disagreement is loud. That's standard defensive engineering, just with a more impressive citation.

### THE TAKEAWAY

If you're running multiple agents against the same codebase, your test suite and static analysis aren't a nice-to-have — they're the only mechanism that turns a silent agent disagreement into something you can actually see and fix before it ships.

## ATLAS: ONE POSTGRES INSTANCE PER MERCHANT

### THE PROBLEM

You're running a multi-tenant database, and every fix for one customer's slow query risks regressing performance for every other customer sharing that instance — to the point where you can't even cleanly attribute a spike to the tenant causing it.

### WHAT THEY DID

Atlas's restaurant-ops platform gives every merchant a dedicated Cloud SQL for PostgreSQL instance instead of a shared multi-tenant database — full isolation and independent scaling, paid for in running far more instances. They moved connection pooling into Cloud SQL Enterprise Plus, killing an entire separate pooling layer and its security surface in the process. Query Insights turned out to be the highest-leverage feature, letting them pin cost and latency spikes to a specific merchant and query instead of guessing. Near-zero-downtime resizing let them scale individual instances without a maintenance window. Result: the database team stopped firefighting plumbing while the business grew 200 to 300% year over year.

### MY TAKE

Dedicated instance per tenant is the simplest isolation model there is — give every customer their own box — and it's the correct trade when engineering time is scarcer than infrastructure budget, which is exactly the position you're in during 200-300% growth. It stops being the right call once tenant count gets large enough that per-instance overhead — patching, backups, monitoring fan-out — outweighs the isolation benefit. At that point you need real multi-tenancy with proper resource governance, not a few hundred copies of the same schema.

### THE TAKEAWAY

Dedicated-instance-per-tenant isn't a permanent architecture, it's a phase. Use it while engineering time is your scarce resource, and start planning the multi-tenant consolidation before instance count — not revenue — becomes your bottleneck.

## COPILOT: ROUTING MODELS WITHOUT BREAKING CACHE

### THE PROBLEM

You're building a multi-model agent system, and every time you switch models mid-conversation to save cost or chase quality, you eat a full context reprocessing cost — the exact tax you were trying to dodge by routing intelligently in the first place.

### WHAT THEY DID

Copilot's two main levers are prompt caching, which reuses model state across repeated prefixes instead of recomputing them on every request, and on-demand tool loading, which lets the model pull in tool schemas only when needed instead of stuffing every definition into context on every turn. The sharper move is in the Auto router: switching models breaks the prompt cache, so it only switches at natural cache boundaries — the first turn, or right after a context compaction — and otherwise sticks with one model to keep the cache warm. Instead of a binary easy-versus-hard task label, the router is trained by scoring weaker against stronger model outputs across quality dimensions, learning where capability actually diverges rather than guessing from a category.

### MY TAKE

Anchoring model switches to cache boundaries is what makes this production-grade instead of a research demo — most multi-model routing writeups ignore that switching has a real, compounding cost, and Copilot built that constraint into the routing decision itself instead of eating the cache miss after the fact. Where this likely breaks down is long sessions with no natural compaction point: run a conversation long enough, and you're stuck on a model that's no longer optimal, because cache locality outranks instantaneous quality in their cost function.

### THE TAKEAWAY

If you're building a multi-model agent system, decide where your cache boundaries live before you design the routing logic. Allow model switches only there, or you'll pay the exact reprocessing cost you built the router to avoid.

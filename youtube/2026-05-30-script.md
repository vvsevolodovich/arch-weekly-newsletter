## COLD OPEN

The entire lakehouse ecosystem spent five years moving catalog metadata out of databases and into scattered JSON files in object storage. Snowflake, Databricks, every major cloud provider bet the farm on this model. DuckLake just reversed it — put all the metadata back in a SQL database — and their reasoning is so obvious in hindsight it's almost embarrassing. If you've ever debugged a corrupt Iceberg table by digging through S3-prefixed JSON files at three in the morning, this one's for you.

## ARTICLE 1 — DuckLake 1.0: Metadata Back in SQL

**Source:** [DuckLake 1.0: Data Lake Format with SQL Catalog Metadata](https://www.infoq.com/news/2026/05/ducklake-sql-catalog/)

---

### THE PROBLEM

You've probably hit this if you've ever run a high-write-rate Iceberg workload. Every transaction generates metadata files — manifest files, manifest lists, snapshot files. At scale you're doing LIST operations on S3 that take seconds. Your catalog server is spending more time tracking files about files than it is tracking actual data. And when you need concurrent writes, you're doing optimistic locking via conditional overwrites on those metadata files, which doesn't scale cleanly under contention. The small-file problem in Iceberg isn't an accident — it's the direct consequence of pushing transactional catalog work onto an object store that was never built for it.

---

### WHAT THEY DID

DuckLake 1.0 puts all catalog and table metadata in a SQL database — Postgres, SQLite, or DuckDB. That's it. The metadata lives in a system designed for exactly this: low-latency, high-concurrency, transactional reads and writes on structured data. They also added data inlining — small inserts and updates land directly in the catalog without creating new Parquet files. High-cardinality columns get bucket partitioning. Filtered queries get sorted tables. Deletion vectors remain wire-compatible with Iceberg readers, so you're not locked in. Version 1.0 ships with backward-compatibility guarantees, which tells you they're serious about production use.

---

### MY TAKE

This is genuinely clever, and the reasoning holds. The industry cargo-culted "metadata in object storage" because it made the storage layer simple and stateless — but it transferred all the complexity onto catalog servers and listing operations that S3 was never built for. DuckLake is basically saying: SQL databases are really good at metadata. We knew this for decades. Let's stop pretending they're not. The real constraint is obvious: your SQL database is now a hard dependency. If it goes down, your lakehouse goes down. You need high-availability Postgres, not a SQLite file. For large enterprises, that's manageable. For a team that wants to avoid ops burden, the irony is you're swapping object-storage complexity for database-availability complexity. Neither is free. But at write-heavy workloads, the SQL approach almost certainly wins.

---

### THE TAKEAWAY

If you're standing up a new internal lakehouse and you're not already locked into a managed Iceberg stack, benchmark DuckLake before you commit. The catalog overhead you're not paying for now compounds painfully at write scale.

## ARTICLE 2 — ClickHouse: 60 Billion Records, 10x Less Storage

**Source:** [How We Reduced Storage 10x While Handling 60+ Billion Records a Day](https://life.wongnai.com/how-we-reduced-storage-10x-while-handling-60-billion-records-a-day-0ed9b9783362)

---

### THE PROBLEM

You've probably hit this if you've tried to correlate a trace with a log during a production incident and found the trace was sampled away. The industry accepted sampling as a necessary cost of observability — keep 1%, maybe 5% of traces, stay within storage budgets. At LINE MAN Wongnai, running Grafana Tempo across three business units, this was the reality: incidents where the trace you needed was the one that got dropped. Worse, the Prometheus plus Tempo stack meant correlating logs and traces across different query interfaces when you're already tired and the production system is on fire.

---

### WHAT THEY DID

They rebuilt the entire observability stack — logs, metrics, and traces — on a single self-hosted ClickHouse cluster. The numbers: 1.5 million rows per second at peak ingest, 143 terabytes of raw telemetry compressed to 14 terabytes on disk — that's a 10x compression ratio — 53% cost reduction, and complete elimination of trace sampling. 100% retention. The mechanism is ClickHouse's columnar storage: analytical queries reading five or six columns across billions of rows pay I/O only for those columns. Row-oriented storage at this scale pays for everything you'll never read. Unifying all three signal types in one store also eliminated the cross-tool correlation overhead of the previous stack.

---

### MY TAKE

The 10x compression number is real — ClickHouse columnar compression at high cardinality on telemetry data is genuinely impressive, and the 100% trace retention is worth calling out. But there's a constraint they underplay: self-hosted ClickHouse is operationally non-trivial. Sharding strategy, replication topology, schema evolution under load — these are problems Grafana Cloud or Datadog solved for you and bundled into a monthly bill. What LINE MAN bought is cost efficiency and data sovereignty at the price of operational complexity. That trade makes sense at their scale — three business units, presumably a dedicated platform team. Below about fifty engineers, I'd think twice. The interesting architecture is the unified observability store itself: once logs, metrics, and traces share a schema, you can write a single SQL query that crosses all three. That's genuinely useful.

---

### THE TAKEAWAY

If you're sampling traces today and keep missing production incidents because of it, the architecture to fix it exists and is cheaper than you think. ClickHouse columnar storage at your scale of data probably costs less than your current Datadog bill. Do the math before you resign yourself to sampling.

## ARTICLE 3 — OpenAI: WebRTC at Global Scale

**Source:** [How OpenAI delivers low-latency voice AI at scale](https://openai.com/index/delivering-low-latency-voice-ai-at-scale/)

---

### THE PROBLEM

WebRTC was designed for peer-to-peer calls and small-scale conferencing. When you try to run it at the scale of a global voice AI product, the conventional full-termination model breaks. One process owns both packet forwarding and all session state — ICE negotiation, DTLS, SRTP key management. That means your load balancer cannot rebalance connections without tearing them down. You can't scale the edge independently from the session logic. And if you want to route users to the nearest point of presence, you can't — because the session state is pinned to wherever it was first established. You're serving a globally distributed user base from wherever your stateful session landed, which is almost never where the user is.

---

### WHAT THEY DID

OpenAI split the relay from the transceiver. The relay is a stateless Go process that lives at the edge, close to the user. It reads packet headers and forwards — nothing else. It doesn't terminate WebRTC. The transceiver owns all the stateful machinery: DTLS, SRTP, ICE negotiation. It scales independently and lives wherever session demands require, not where the user happens to be. The relay distributes globally for fast connection setup and low round-trip time. The transceiver handles the expensive crypto and negotiation wherever there's capacity. The scaling curves are separated: edge latency and session-state complexity no longer have to trade off against each other.

---

### MY TAKE

This is the correct decomposition. The relay-transceiver split follows the same principle as separating hot path from stateful path in any distributed system — it's clean, it's principled, and the relay being truly stateless is the elegant part. It doesn't even need to understand what it's forwarding, just where to send it. What they don't discuss is transceiver failover. When a transceiver goes down mid-call, where does the session state go? Active-passive transceiver failover with DTLS state checkpointing is non-trivial, and I'd want to understand that failure mode before I called this fully solved. The piece also doesn't quantify the latency improvement at the 99th percentile — which is what a voice user actually experiences when the system is under load. I suspect those numbers exist internally and are good, but I'd want to see them.

---

### THE TAKEAWAY

If you're building any real-time media product at scale — voice, video, gaming — the stateless-forwarder pattern is the right abstraction boundary. Don't let session state infect your edge layer. Separate the two from the start.

## ARTICLE 4 — Google Virgo: Networking 134,000 TPUs for AI

**Source:** [How we evolved Google's global and data center networks for the AI era](https://cloud.google.com/blog/products/networking/data-center-and-global-networks-built-for-ai-era)

---

### THE PROBLEM

AI training clusters have a fundamentally different network requirement than anything the data center world was built for. Traditional distributed workloads care about average latency and reasonable aggregate bandwidth. A transformer training job on a hundred thousand accelerators needs all-reduce operations to complete synchronously — which means your bandwidth constraint is your slowest node, and a single bad link poisons the entire batch. The penalty for imbalanced bandwidth isn't degraded performance. It's a training run that simply does not complete. Getting to exaflop-class training requires rethinking the fabric from the ground up, not just adding more switches.

---

### WHAT THEY DID

Google's Virgo Network uses a flat two-layer non-blocking topology with high-radix switches. A single fabric connects 134,000 TPU 8t chips at 47 petabits per second of bisectional bandwidth — 4x per-accelerator versus the prior generation, with 40% lower unloaded latency. The network also ships sub-millisecond telemetry, automated straggler detection, and per-domain fault isolation designed for beyond-nines availability without system-wide restarts. Then there's the piece they mention almost as an aside: the real constraint is power distribution, not data movement. Electrons are harder to route than photons. So compute clusters locate near power sources, and Google's AI-native Cloud Interconnect moves data cheaply at 3.2 terabits per second — dropping petabyte transfer time from 22 hours to 42 minutes.

---

### MY TAKE

The bandwidth numbers are staggering but expected — Google has the budget for it, and the TPU roadmap has always been vertically integrated in ways that give them architectural advantages here. What I find genuinely interesting is the straggler detection, mentioned almost in passing. At 134,000 chips, your 99th percentile node is your constraint on every single batch. Automated straggler detection and fault isolation — without a system-wide restart — is where the real reliability engineering lives, and it's a harder problem than the headline bandwidth. The power constraint observation is also underappreciated across the industry: the limiting resource for AI infrastructure is increasingly electrical capacity. You can buy more GPUs. You cannot buy more grid capacity. That shapes where training clusters physically get built for the next decade.

---

### THE TAKEAWAY

When evaluating cloud providers for large model training, ask specifically about per-accelerator bisectional bandwidth and straggler detection policy. The GPU count headline is marketing. The networking topology and automatic fault isolation are what determine whether your multi-day training run actually completes.

## ARTICLE 5 — Netflix: Killing 3 PB of Cassandra Pipeline Debt

**Source:** [The Evolution of Cassandra Data Movement at Netflix](https://netflixtechblog.medium.com/the-evolution-of-cassandra-data-movement-at-netflix-6e13329c80a1)

---

### THE PROBLEM

You've probably built a pipeline that worked and then kept working until it didn't. Netflix's Casspactor ran at 1,200 batch movements per day and shifted 3 petabytes of data from Cassandra into Iceberg tables. It also accumulated years of structural debt: metadata fell out of sync with actual backups, multiple intermediate Iceberg stages bloated storage, and a fragile multi-service dependency chain silently corrupted results. That last one is the worst kind of failure — not a crash, not an error, but a wrong answer that takes months to notice. The system looked healthy. The data wasn't.

---

### WHAT THEY DID

They simplified radically. The replacement reads Cassandra backup files directly from S3 and generates Spark DataFrames in a single pass — no intermediate tables, no multi-stage connector chain. The entire intermediate stack is gone. An auto-sizer adjusts job resources based on source table characteristics without manual tuning. The redesign is governed by three principles they call non-negotiable: Validation — confidence in data consistency — Visibility — real-time instrumentation — and Safety — minimizing user impact during transition. This system now sits inside Netflix's Data Bridge platform, which orchestrates 20,000 distinct job types and 300,000 executions per week. That context matters: the platform needs to be operationally simple, or it breaks at scale.

---

### MY TAKE

The most important pattern here is not the technical specifics of the S3-to-Spark pipeline — it's the observation that silent corruption is the worst failure mode, and the cure is radical simplification plus real-time validation. The intermediate stages were added for good reasons originally — they probably made individual stages easier to debug and test in isolation. But they created emergent complexity: the system as a whole became harder to reason about than any individual stage. Single-pass pipelines are harder to build correctly, but much easier to verify end-to-end. The auto-sizer is also worth stealing. Manual job tuning at 300,000 executions per week is an operational liability that compounds every time someone leaves the team who understood the tuning decisions.

---

### THE TAKEAWAY

If your data pipeline has more than three stages, draw the full dependency graph. Count the places where data can be wrong and nobody will notice for a week. Then ask honestly whether each intermediate stage is making the system simpler or more complex in aggregate. You will probably surprise yourself.

## ARTICLE 6 — GitHub Issues: From 2% to 35% Under 200ms

**Source:** [From Latency to Instant: Modernizing GitHub Issues Navigation Performance](https://github.blog/engineering/architecture-optimization/from-latency-to-instant-modernizing-github-issues-navigation-performance/)

---

### THE PROBLEM

At the start of 2026, only 2% of GitHub Issues page views completed in under 200 milliseconds. Two percent. On a product used by essentially every software engineer on the planet, navigating between issues felt slower than a filing cabinet. The baseline expectation has shifted: local-first tools like Linear have trained engineers to expect navigation that feels instant. GitHub was losing the perception battle against its own competitors — teams that built faster issue trackers using GitHub to host the code that ran those trackers.

---

### WHAT THEY DID

The fix layers a client-side cache backed by IndexedDB with a preheating strategy. A service worker proactively walks high-intent issue references and pre-warms cache entries before the user navigates. When you click a link, the content is already local. The page renders from cache first, then revalidates in the background. This pushed 35% of issue views under 200ms — from 2% to 35%. The team is now moving toward an edge delivery layer closer to users and backend rewrites optimized explicitly for low-latency delivery.

---

### MY TAKE

The approach is sound and the results are real — 2% to 35% is not a rounding error. But I want to push back on framing IndexedDB preheating as the destination. It's a client-side workaround for a slow backend, and it has failure modes that scale with your user base: cache invalidation bugs, stale content, users on different devices or private browsing sessions who get none of this. The preheating strategy only works if you correctly predict which issues the user will navigate to, and that prediction degrades for non-linear browsing patterns. The real fix is what they mention at the end — backend rewrites optimized for low-latency delivery. That said: 2% to 35% is a win that shipped in weeks. You don't ignore that. The service worker bought them time and real data on where the backend needs to improve.

---

### THE TAKEAWAY

If your web app is slow and you're waiting for the backend rewrite, client-side preheating with a service worker is a legitimate intermediate step. Just don't let it delay the actual fix, and instrument the cache miss rate so you know when preheating stops working.

## [How Stripe uses graph search and state machines to auto-remediate a global database fleet](https://stripe.dev/blog/how-stripe-uses-graph-search-and-state-machines-to-auto-remediate-a-global-database-fleet) 🤓

Stripe's MongoDB infrastructure spans 2,000+ shards handling 5 million QPS, and on-call engineers previously diagnosed and manually sequenced recovery operations — a process that generated ~200 alert pages per year and left shards in unhealthy states for a cumulative 12 days annually. The solution models the entire shard topology as a traversable graph and uses BFS-based pathfinding to compute the minimum-length sequence of valid state transitions from any bad state to healthy, with a finite state machine enforcing transition legality at each step. New shard layouts require zero changes to recovery logic because remediation derives from the graph rather than hard-coded procedures. The automated system reduced pager volume by 30% and eliminated all 12 accumulated days of annual unhealthy shard time. #db #distributed #architecture #engineering

## [GitFarm: Git® as a Service for Large-Scale Monorepos](https://www.uber.com/us/en/blog/gitfarm-as-a-service/) 🤓

Cloning Uber's multi-gigabyte monorepo consumed up to 15 minutes of initialization time and proportional compute on every automation system that touched it — GitFarm solves this by serving Git as a Service over gRPC from pre-warmed bare repository clones maintained as ephemeral sandboxes. A client calls any standard Git command via API; GitFarm pulls a sandbox from its warm pool in under 500ms, executes the operation in isolated ephemeral storage, and tears it down on completion, enabling multi-step workflows like code-review validation and audit PR generation without persistent local state. The platform cuts resource overhead by 80%+ and client-side resource utilization by over 70%, reducing initialization from up to 15 minutes to under 1 minute and removing a systemic bottleneck that made large-scale monorepo automation impractical. #softwareengineering #engineering #development #architecture

## [Cost Attribution in Discord's API](https://discord.com/blog/cost-attribution-in-discords-api) 👨‍💼

Discord's unified Python API backend powers 1,700+ endpoints and 700 background tasks across hundreds of Kubernetes deployments, but had no way to answer "what does it cost to run the messaging feature?" until they introduced feature groups — logical clusters of endpoints and tasks (e.g., "chat" groups all send/view/edit message operations and fan-out tasks) that serve as the cost attribution unit. A 10 Hz CPU-time sampling profiler tags samples with feature-group labels and pipes results into Pyroscope, which then joins against cloud billing data to produce per-feature cost breakdowns across the engineering org. Measuring at CPU-time granularity per logical feature — not per request or endpoint — is what makes the model accurate for monoliths where many features share the same code paths. #softwareengineering #engineering #devops #architecture

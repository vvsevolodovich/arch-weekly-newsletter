## [Optimizing vector search using Amazon S3 Vectors and Amazon OpenSearch Service](https://aws.amazon.com/blogs/big-data/optimizing-vector-search-using-amazon-s3-vectors-and-amazon-opensearch-service/) 👨‍💼

S3 Vectors' GA integration with Amazon OpenSearch, announced July 9, materializes the "storage-first" architecture that decouples compute from storage for RAG workloads — a design that reduces the cost of storing and querying vectors by up to 90% while maintaining sub-100ms query performance. Two integration paths address different latency budgets: exporting S3 Vectors data to OpenSearch Serverless targets 10ms latency for high-throughput search, while using S3 Vectors as a cost-optimized storage engine inside OpenSearch trades some performance for dramatically lower cost at idle. Per-index capacity has increased 40× to 2 billion vectors, removing the most common scale barrier for teams trying to move from RAG prototypes into production. #db #cloud #architecture #ai

## [Graviton5 CPU Benchmarks: 30% Geo Mean Improvement Over Graviton4](https://www.phoronix.com/review/aws-graviton5) 🤓

Phoronix's independent testing of Graviton5 M9g against Graviton4 M8g shows a 30% geo-mean improvement, attributable to the jump from Neoverse-V2 to Neoverse-V3 cores, a 2.8 → 3.3 GHz clock increase, 12-channel DDR5-8800 memory (replacing DDR5-5600), 5× larger L3 cache (192 MB), and PCIe Gen 6 — all on TSMC's 3nm node at 192 cores per chip. Customer production data reinforces the benchmarks: HubSpot saw MySQL query duration fall by 60%, Honeycomb observed 36% more throughput per core over a 6-month A/B test, and ClickHouse logged 36% better performance with no code changes. For most CPU-bound cloud workloads outside of x86-specific instruction dependencies, the cost-to-performance calculus has shifted firmly to Arm, and M9g represents the clearest expression of that gap yet. #cloud #engineering #softwareengineering

## [Experiences with local models for coding](https://martinfowler.com/articles/exploring-gen-ai/local-models-for-coding-experiences.html) 👨‍💼

Four weeks of hands-on agentic coding tests on M3 Max (48 GB) and M5 Pro (64 GB) hardware confirms the pattern the viability analysis predicted: capable local models handle short-context, single-file tasks reasonably well but degrade sharply on multi-step agent loops, with tool-calling failures being the most consistent breakage point rather than raw model quality. The key operational finding is that models that fully fit in RAM are the only viable candidates — anything that forces paging produces latency that makes agentic loops impractical regardless of output quality. Teams evaluating on-device model stacks for code agents should expect to invest heavily in harness engineering to compensate for the reliability gap before reaching production-grade autonomous loop performance. #ai #llm #development #engineering

## [Scaling Real-Time Traffic Forecasting with a Graph-Aware Transformer](https://www.uber.com/us/en/blog/scaling-real-time-traffic/) 🤓

Uber's DeepETT replaces a statistical ETA predecessor with a deep graph-aware transformer jointly trained on road-network topology and real-time traffic signals, now running at 2 million forecasts per second across global markets — one of Uber's highest-throughput deep learning deployments. Head-to-head benchmarks show 6% better long-trip arrival time accuracy and a 19% boost in forecast variance explained; the team estimates roughly $100M in incremental annualized revenue from tighter routing and pickup matching. Scaling a spatially-structured deep model to this throughput required purpose-built graph data pipelines that encode adjacency structure into the transformer without blocking production latency budgets. #ai #architecture #distributed #engineering

## [Introducing Dimster, a Performance Benchmarking Tool for Apache Kafka](https://jack-vanlightly.com/blog/2026/5/20/introducing-dimster-a-performance-benchmarking-tool-for-apache-kafka) 👨‍💼

Existing tools like OpenMessagingBenchmark run single-point tests; Dimster centers its design on dimensional testing — automatically sweeping configuration axes such as batch.size or consumer type across a grid of test points within one campaign, then emitting self-contained result bundles that include performance data, effective configs, hardware metadata, logs, and charts for every point. The tool is Kubernetes-native: a laptop CLI speaks k8s and orchestrates runs on any hardware or cloud with identical execution logic, so results are reproducible across environments. For teams evaluating Kafka broker variants, validating tuning decisions, or sizing new clusters, Dimster provides the comparative rigour that ad-hoc benchmarking cannot. #distributed #engineering #quality #architecture

## [Performance Engineering in Distributed Systems: Lessons That Compound Over Time](https://cacm.acm.org/blogcacm/performance-engineering-in-distributed-systems-lessons-that-compound-over-time/) 👨‍💼

Performance failures in large-scale distributed systems rarely stem from a single dramatic flaw; they compound from accumulated micro-decisions about hot-path behavior, data encoding, request fan-out, and traffic distribution — each benign alone, devastating together. The prioritized framework for intervention proposed here runs: reduce unnecessary work first, then fix data layout and representation, then shift expensive computation earlier in the pipeline, then optimize for actual workload shape, then tune infrastructure behavior. Written from production experience across multiple large systems, the piece distills into a practitioner checklist that applies across languages and stack types before investing in larger architectural changes. #distributed #softwareengineering #quality #engineering

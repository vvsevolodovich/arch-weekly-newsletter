## [Running a self-hosted LLM in Kubernetes with vLLM](https://www.cncf.io/blog/2026/07/16/running-a-self-hosted-llm-in-kubernetes-with-vllm/) 👨‍💼

Cost predictability at high request volumes, tighter per-request latency control, and data-residency obligations that bar prompt data from leaving a specific jurisdiction are the three structural reasons teams choose to run LLMs in-house rather than route through managed provider APIs. The post walks through deploying vLLM — the dominant open-source serving runtime for transformer models — as a first-class Kubernetes workload, covering GPU node configuration, model loading, and horizontal scaling under standard Kubernetes scheduling and resource policies. For organizations already operating Kubernetes, the incremental operational overhead is modest relative to the compliance and cost control benefits, particularly when token volumes make per-call cloud pricing untenable. #llm #cloud #devops #architecture

## [Running AI Agents Safely Inside Kubernetes, 2026 Field Guide](https://kodekloud.com/blog/running-ai-agents-safely-inside-kubernetes/) 👨‍💼

Of all the security controls for AI agents deployed on Kubernetes, network egress is the single highest-impact layer: agents that can reach arbitrary external endpoints negate isolation established by RBAC and namespace boundaries. Any agent that executes generated code — tool-calling, code-interpreter, or sandboxed evaluation patterns — requires a dedicated sandboxed runtime; running it as an ordinary container is architecturally insufficient. The next generation of isolation is already entering mainstream clusters: WebAssembly runtimes offer near-VM security boundaries with container-level cold-start times, and confidential containers backed by Intel TDX or AMD SEV-SNP provide hardware-enforced memory encryption for the most sensitive agent workloads. #cloud #devops #architecture #softwareengineering

## [Navigating the ingress-NGINX Retirement](https://www.cncf.io/blog/2026/07/09/navigating-the-ingress-nginx-retirement/) 👨‍💼

The community-maintained ingress-nginx controller reached end-of-life in March 2026 — not because of usage decline, but because maintainer capacity collapsed under the weight of an inherently unsafe feature: arbitrary NGINX snippet annotations that allowed any developer to inject raw NGINX directives into the global configuration. Teams now face two migration paths: a lift-and-shift swap to an Envoy-based controller like Contour (faster to execute, but every nginx.ingress.kubernetes.io/* annotation must be manually translated or rebuilt using Contour's CRDs) or adopting Kubernetes Gateway API, which enforces explicit role separation between infrastructure operators and application developers and is now the upstream-endorsed successor to Ingress. Using this forced migration as a lever to move to Gateway API is the architecturally correct choice for most organizations — the Ingress API itself remains stable; only the nginx implementation is gone. #cloud #devops #architecture #engineering

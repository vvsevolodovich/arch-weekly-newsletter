## [How we evolved Google's global and data center networks for the AI era](https://cloud.google.com/blog/products/networking/data-center-and-global-networks-built-for-ai-era) 🤓

Google's Virgo Network uses a flat, two-layer non-blocking topology with high-radix switches to eliminate the bandwidth tax on AI training clusters: a single fabric connects 134,000 TPU 8t chips at 47 petabits/sec of bisectional bandwidth, 4× per-accelerator versus the previous generation with 40% lower unloaded latency. The harder constraint is power distribution, not data movement—electrons are harder to route than photons, so compute clusters near power sources and the AI-native Cloud Interconnect moves data cheaply at 3.2 Tbps increments, dropping petabyte transfer time from 22.2 hours to 0.7 hours. Sub-millisecond telemetry, automated straggler detection, and per-domain fault isolation target "beyond-nines" availability without system-wide restarts across 43 cloud regions and 200+ edge locations. #cloud #architecture #distributed #engineering

## [Introducing Google AI Threat Defense to help you outpace the adversary](https://cloud.google.com/blog/products/identity-security/introducing-google-ai-threat-defense) 👨‍💼

Google AI Threat Defense composes Gemini reasoning, Wiz's contextual risk graph, CodeMender's automated fix generation, and Mandiant intelligence into a four-step pipeline (Prepare → Scan → Remediate → Monitor) that takes a vulnerability from discovery to production patch without manual intervention. The urgency: exploitation windows have collapsed from weeks to hours, while traditional remediation stretches to months—the platform claims to compress patching to minutes by auto-generating and test-validating code fixes before deployment. Cost architecture is tiered: lightweight proxy models run continuous scanning, frontier models activate only for high-risk findings, built on infrastructure that already blocks 10 million spam emails per minute. #devops #cloud #security #engineering

## [GPU autoscaling on Kubernetes with KEDA: Building an external scaler](https://www.cncf.io/blog/2026/05/27/gpu-autoscaling-on-kubernetes-with-keda-building-an-external-scaler/) 👨‍💼

Kubernetes' HPA reasons about CPU and memory—neither reflects the true saturation signal for GPU inference workloads running vLLM, Triton, or agentic inference stacks, causing wasted accelerator capacity and degraded latency. KEDA's core binary compiles with CGO_ENABLED=0, making direct NVML bindings impossible; the fix is a custom external gRPC scaler deployed as a DaemonSet that reads GPU utilization, memory pressure, temperature, and power draw natively from NVML C-bindings without requiring a Prometheus stack. The implementation supports scale-to-zero and frames idle GPU capacity as a GreenOps problem—wasted cycles translate directly to wasted energy and higher Scope 3 emissions. #devops #cloud #engineering #ai

## [How we built Cloudflare's data platform and an AI agent on top of it](https://blog.cloudflare.com/our-unified-data-platform/) 👨‍💼

Cloudflare's Town Lake is a data lakehouse built entirely on Cloudflare's own stack — Apache Trino as the query engine joins Postgres, ClickHouse, and Iceberg-on-R2 tables in a single SQL statement without intermediate materialization, with Workers handling compute and Workflows handling orchestration. Skipper, the AI agent sitting on top, routes through Anthropic Claude models treated as commodity APIs and uses TypeScript reachability analysis plus call-graph mapping to validate generated tool calls before execution. In a recent measurement period, 324 Cloudflare employees issued 91,760 queries — 53% billing-related — making the business case for centralizing analytical access instead of maintaining per-domain data silos. #architecture #cloud #ai #engineering

## [Open House 2026 Day 1: Real-time data without lock-in and what teams can build next](https://clickhouse.com/blog/open-house-2026-day-1) 🤓

ClickHouse Cloud's multi-stage distributed query execution (private preview) repartitions intermediate data between query stages across any available worker — viable because the shared-storage architecture carries no fixed shard ownership. Early TPC-H SF100 benchmarks show the full suite dropping from 117.6 seconds on one node to 54.7 seconds across eight, with a cost-based optimizer still maturing for further gains. ClickHouse Postgres entered public beta delivering over 5× the TPS of AWS RDS; write support landed for Microsoft OneLake and Unity Catalog, enabling bi-directional lakehouse interoperability. #db #distributed #architecture #engineering

## [Kafka Share Groups and Parallelizing Consumption — Part 2: Producer Batches and share.acquire.mode](https://jack-vanlightly.com/blog/2026/5/27/kafka-share-groups-and-parallelizing-consumption-part-2-producer-batches-and-shareacquiremode) 🤓

Kafka Share Groups — production-ready in Kafka 4.2.0 — allow consumers to parallelize beyond partition count, but the speedup depends on two configuration knobs: max.poll.records (consumer-side cap per poll) and group.share.partition.max.record.locks (broker-side cap on in-flight records per partition). Part 2 of this Dimster-based benchmarking series examines producer batch sizes and share.acquire.mode (prefetch vs sequential), finding that simulating realistic processing time substantially changes the winning configuration compared to raw-throughput tests alone. Share group overhead versus traditional consumer groups remains small, validating the feature as a production-viable consolidation point for task-queue workloads that previously required separate messaging infrastructure. #distributed #architecture #engineering

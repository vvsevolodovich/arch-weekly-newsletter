## [Announcing Durability for Amazon ElastiCache for Valkey](https://aws.amazon.com/blogs/database/announcing-durability-for-amazon-elasticache/) 👨‍💼

Amazon ElastiCache for Valkey now ships durable storage as a first-class option, with two modes: write-ahead logging that minimizes data loss at the cost of write overhead, and a replication-backed option that accepts a narrow data-loss window in exchange for lower latency. The architectural significance is that teams previously had to maintain an ephemeral cache alongside a separate persistent store; this feature collapses those two tiers for workloads where occasional read-after-write lag is acceptable. AWS is positioning Valkey — the open-source Redis fork created after Redis relicensed in 2024 — as the durable, fully managed cache-plus-store for its ecosystem. #db #cloud #architecture

## [Build Secure, AI-Driven Workflows with Terraform and Vault MCP Servers](https://www.hashicorp.com/en/blog/build-secure-ai-driven-workflows-with-new-terraform-and-vault-mcp-servers) 🍼

HashiCorp's Terraform and Vault MCP Servers create a credential-safe bridge between AI assistants and infrastructure tooling: the Terraform server gives agents real-time access to provider docs, module metadata, and HCP Terraform workspace state, while the Vault server provides auditable secrets management — all with the MCP server inheriting the operator's auth tokens so the LLM layer never sees raw credentials. The security design is the core insight: the MCP server is the trust boundary, meaning prompt injection cannot escalate privileges beyond what the session account already holds, making this the first formally auditable interface between AI coding agents and secrets infrastructure. Both servers work with Claude, Cursor, Gemini, and similar assistants. #devops #cloud #ai #engineering

## [GitHub Availability Report: May 2026](https://github.blog/news-insights/company-news/github-availability-report-may-2026/) 👨‍💼

Nine incidents in May 2026, but the technically revealing one hit on May 6: a 32-bit integer in a Vitess lookup table for PR review threads exhausted its address space, causing near-100% failure of all new thread creation for 3 hours 50 minutes. The root cause is a canonical sharding hazard — the primary table had already been migrated to 64-bit, but the secondary Vitess lookup table had not, and no alerting caught the drift. AI coding agent workloads are driving write rates that GitHub's integer key spaces weren't provisioned to handle; GitHub responded by raising its capacity-planning target from 10x to 30x. #engineering #distributed #softwareengineering

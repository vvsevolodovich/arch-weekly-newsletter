## [The Evolution of Cassandra Data Movement at Netflix](https://netflixtechblog.medium.com/the-evolution-of-cassandra-data-movement-at-netflix-6e13329c80a1) 🤓

Netflix's Casspactor pipeline ran at ~1,200 batch movements per day and shifted 3 PB of data from Apache Cassandra into Iceberg tables, but accumulated structural debt: metadata fell out of sync with actual backups, connectors produced multiple intermediate Iceberg stages, and a fragile multi-service dependency chain silently corrupted results. The replacement reads Cassandra backup files directly from S3 and generates standard Spark DataFrames in a single pass — eliminating the entire intermediate-table stack and the storage bloat it caused — while an intelligent auto-sizer adjusts job resources based on source table characteristics without manual tuning. The redesign is governed by three non-negotiable pillars: Validation (confidence in data consistency), Visibility (real-time instrumentation), and Safety (minimizing user impact), applied across Netflix's Data Bridge platform which now orchestrates 20,000 distinct job types and 300,000 executions per week. #db #distributed #architecture #engineering

## [Making User-Sequence Data More Cost-Efficient, Faster, and Easier to Use](https://medium.com/pinterest-engineering/making-user-sequence-data-more-cost-efficient-faster-and-easier-to-use-2a56a928cae1) 👨‍💼

Pinterest replaced a proliferation of bespoke, per-signal data pipelines with a unified lambda architecture for user-sequence data: a streaming path maintains near-real-time views for online inference while a batch path handles backfills and long-sequence recomputation from raw historical data. Adding a new event type or enrichment shifted from standing up a new pipeline to writing a small composable executor and a config change, compressing onboarding time substantially. Bucketing and local sorting by UserId compresses sequence tables to as little as 25% of original size via Delta Encoding on sequential features, and major recommendation surfaces recorded measurable engagement gains after migrating to the new platform. #architecture #softwareengineering #engineering #ai

## [DuckLake 1.0: Data Lake Format with SQL Catalog Metadata](https://www.infoq.com/news/2026/05/ducklake-sql-catalog/) 🤓

DuckLake 1.0 inverts the metadata design of Iceberg and Delta Lake: instead of scattered per-transaction files in object storage, all catalog and table metadata lives in a SQL database (PostgreSQL, SQLite, or DuckDB), exploiting decades of database optimization for low-latency, high-concurrency catalog operations. The SQL-catalog design directly addresses the small-file problem and catalog-listing overhead that degrades large-scale Iceberg deployments under high write rates. Version 1.0 ships production-ready with backward-compatibility guarantees and introduces data inlining (small inserts/updates land in the catalog without creating new Parquet files), bucket partitioning for high-cardinality columns, sorted tables for filtered-query acceleration, and deletion vectors wire-compatible with Iceberg readers. #db #architecture #distributed #softwareengineering

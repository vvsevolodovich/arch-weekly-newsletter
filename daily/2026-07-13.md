## [Getting started with Change Data Capture in Amazon Aurora DSQL](https://aws.amazon.com/blogs/database/getting-started-with-change-data-capture-in-amazon-aurora-dsql/) 👨‍💼

Aurora DSQL's native CDC — now at GA — closes the event-driven gap in AWS's serverless distributed SQL service: every committed INSERT, UPDATE, and DELETE is serialized into a structured JSON record and delivered to a Kinesis Data Stream at at-least-once semantics with zero measured impact on database performance. The integration uses a bring-your-own-target model where you provision the Kinesis stream and DSQL assumes an IAM role to write CDC records, cleanly decoupling the database from any downstream consumer without introducing proprietary streaming abstractions. Downstream, the same Kinesis stream fans to Lambda for real-time processing, Firehose for durability-first delivery to S3, Redshift, or OpenSearch, or any custom consumer — making this the canonical integration point for DSQL-backed event-sourcing patterns that previously required application-layer polling. #db #distributed #architecture #softwareengineering

## [Google Cloud Run sandboxes are in public preview](https://cloud.google.com/blog/topics/developers-practitioners/google-cloud-run-sandboxes-are-in-public-preview) 👨‍💼

Cloud Run Sandboxes solve the structural problem of running AI-generated code safely inside an existing service instance: sandboxes spawn within your container's allocated resources in milliseconds at no additional cost, and a fleet of 1,000 concurrent sandboxes averages 500ms end-to-end latency. The isolation enforces three hard boundaries simultaneously — credential isolation (no access to the service's env vars or GCP metadata server), deny-all network egress by default, and a read-only filesystem overlay where all writes land in a temporary memory-backed layer discarded after execution — achieving defense-in-depth without dedicated microVM infrastructure or premium pricing. The API surface is a subprocess call wrapping `sandbox do`, giving teams a near-zero-migration path for today's dynamic code interpreter workloads with native ADK integration for agent workflows and an explicit flag to selectively open egress when needed. #cloud #softwareengineering #ai #devops

## [Apache Kafka performance #1 - linger.ms](https://jack-vanlightly.com/blog/2026/7/7/apache-kafka-performance-1-lingerms) 🤓

`linger.ms` — the client-side producer setting that delays batch dispatch to allow more records to accumulate — has workload-dependent behavior that makes a universal default a trap: at 5K records/sec with a ~24ms inter-arrival gap, `linger.ms=5` adds latency without forming larger batches because the timer fires before the next record arrives, netting zero benefit. At 100K records/sec (roughly 833 records per producer per second), `linger.ms=20` showed a massive improvement over both 0 and 5ms because at that throughput a producer consistently fills a batch before the linger timer expires. The practical takeaway is that linger should approximate the inter-arrival time of records at the configured partition, not at total topic throughput — a distinction that matters most in keyed workloads where per-partition rate is a small fraction of overall producer send rate. #distributed #architecture #engineering #softwareengineering

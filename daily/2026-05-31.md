## [Under the River](https://shopify.engineering/under-the-river) 👨‍💼

River is Shopify's internal AI agent that operates exclusively in public Slack channels — a hard architectural constraint that doubles as a transparency mechanism, since every prompt, commit, and code review happens where any employee can observe it. In a recent 30-day window River ran 59,918 sessions across 5,170 channels, touching the work of 7,000+ people, and coauthored 3,536 merged pull requests — reaching 1 in 8 of all merged PRs company-wide. The agent runs on Aquifer, Shopify's internal agent platform handling session management, sandboxed execution, a durable event log, credentials proxying, and observability; River itself is just a configuration profile on top. The practical lesson: making AI work visible drove adoption faster than any capability announcement. #ai #engineering #softwareengineering #architecture

## [The Kubernetes Integration Tax: Prometheus, Cilium and Production Reality](https://www.cncf.io/blog/2026/05/28/the-kubernetes-integration-tax-prometheus-cilium-and-production-reality/) 👨‍💼

Platform teams typically spend 80% of their time wiring CNCF projects together rather than installing or tuning them — the author calls this the "integration tax." The concrete failure: Prometheus had no ServiceMonitors wired to Cilium's agent and operator pods, making two correctly-installed tools completely invisible to each other and leaving Grafana dashboards blank during a 2 AM incident. The CNCF landscape has ~250 projects but production clusters standardize on 20–30 core tools, and every Helm chart upgrade reintroduces integration surfaces. The proposed fix is a codified platform repo with Cilium NetworkPolicies and Prometheus ServiceMonitors pre-wired into every chart, ArgoCD propagating validated configurations cluster-wide via version bumps — turning per-cluster glue work into a one-time, auditable platform investment. #devops #cloud #engineering #architecture

## [The Hugo Evolution: Engineering Grab's Unified, One-Click Data Ingestion Platform with Apache Flink](https://engineering.grab.com/one-click-data-ingestion-platform-with-apache-flink) 👨‍💼

Grab's data ingestion previously split across Sprinkler (Spark batch) and Kafka Connect (streaming), requiring teams to mentally map concepts across two tool surfaces — raising onboarding friction and coordination overhead between platform and product teams. Hugo's redesign consolidates both onto Apache Flink: a single click-to-query UI automatically orchestrates Flink consumption, S3 writes, and Spark compaction, hiding multi-stage pipeline complexity from users. Schema evolution is now zero-touch — Flink pipelines retrieve Protobuf schemas from Confluent Schema Registry at startup, and Hugo detects changes, validates compatibility, and updates Hive tables without manual intervention. Flink is currently the new default for Kafka-source pipelines, with an explicit migration path for teams still on Kafka Connect. #distributed #architecture #engineering #softwareengineering

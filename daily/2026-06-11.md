## [AI Serving Platform That Adapts to Your Model](https://www.databricks.com/blog/ai-serving-platform-adapts-your-model) 👨‍💼

Serving a 2 MB scikit-learn classifier on one CPU core and a fine-tuned 70B LLM on eight GPUs from the same platform — without manual resource configuration — is the central claim of Databricks' new AI Serving Platform, which delivers 300K+ QPS at sub-10ms p99 latency overhead and up to 90% lower infrastructure cost versus self-managed stacks. Every model endpoint is a fully isolated Kubernetes deployment so that one endpoint's traffic spikes, failures, or resource pressure cannot bleed into others — a security boundary as much as a reliability one. The architectural bet is adaptive resource allocation: continuously reading traffic patterns and model characteristics to find the optimal cost-performance curve rather than over-provisioning to peak. #ai #cloud #engineering

## [The Inference Paradox: How Split-Brain LLMs Are Killing Your GPU ROI](https://kubex.ai/blog/the-inference-paradox-how-split-brain-llms-are-killing-your-gpu-roi/) 🤓

LLM inference has a structural hardware mismatch: the prefill phase is compute-bound (processing all input tokens in a single forward pass) while decode is memory-bandwidth-bound (reading the full KV cache to emit one token per step), so coupling both phases on the same GPU means each permanently starves the other. Kubex's enterprise audits surface average GPU utilization near 5% — monolithic vLLM serializes all prefill before decode can continue, and under high concurrency the delay compounds across every request in the batch. Disaggregating prefill and decode onto separate hardware-optimized node pools — as llm-d (now a CNCF sandbox project) implements on Kubernetes with prefix-cache-aware routing — yields 2–3x throughput at high concurrency by keeping the decode pool continuously active while the prefill pool handles bursts in parallel. #ai #llm #engineering #cloud

## [The API portal is the clearest signal of whether your company can handle AI agents](https://thenewstack.io/mcp-api-governance-readiness/) 👨‍💼

The quality of your existing API governance stack — OpenAPI specs, developer portals, access control, rate limiting — directly determines how fast you can surface capabilities to AI agents via MCP. Under the hood, MCP is just a long-lived HTTP connection serving JSON; the real constraint is whether you already have machine-readable API definitions and consistent permission boundaries that agents can discover and invoke safely. Enterprises that invested in governance infrastructure — documented APIs, AsyncAPI contracts, fine-grained security policies — are finding those investments were preparation for exactly this transition, while organisations sitting on undocumented REST sprawl have a deeper remediation problem than any SDK can solve. #architecture #ai #softwareengineering #engineering

## [Rearchitecting the Workflows control plane for the agentic era](https://blog.cloudflare.com/workflows-v2/) 🤓

Cloudflare's Workflows V2 breaks the concurrency ceiling that V1's monolithic Account coordinator imposed by splitting control-plane responsibilities into two primitives: SousChef tracks lifecycle metadata for a per-account subset of workflow instances, and Gatekeeper distributes concurrency "slots" across all SousChefs — separating metadata management from execution. Each workflow step is now isolated, replayable, and idempotent, so a worker restart resumes from the last committed step checkpoint rather than re-running the full workflow from scratch. The result is an 11x increase in concurrent instances (4,500 → 50,000), 3x higher creation rate (100 → 300 starts/sec per account), and 2M queued instances per workflow — sufficient headroom to run multi-step AI agent pipelines at cloud scale without building a custom orchestration layer. #distributed #architecture #cloud #ai

## [Rethinking Distributed Systems for Serverless Performance and Reliability](https://www.databricks.com/blog/rethinking-distributed-systems-serverless-performance-and-reliability) 🤓

Spark Connect is the architectural break that makes Databricks serverless feasible: a gRPC client-server split moves user application code out of the Spark driver process, shifting the unit of execution from application processes to individual queries and eliminating the failure cascade where one workload's memory exhaustion can crash adjacent jobs sharing a cluster. A Serverless Gateway handles multi-tenant workload routing across compute resources, and an adaptive autoscaler continuously right-sizes cluster capacity for both cost and latency targets. Together these layers deliver 25+ major Spark runtime upgrades per year at a 99.998% success rate across 4.5 billion workloads, transparently, with zero user action required. #distributed #architecture #cloud #softwareengineering

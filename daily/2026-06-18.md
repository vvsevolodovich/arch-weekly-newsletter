## [How Atlas Scales Hundreds of Cloud SQL Databases](https://cloud.google.com/blog/products/databases/how-atlas-scales-hundreds-of-cloud-sql-databases) 👨‍💼

Atlas gives every merchant on its restaurant-ops platform a dedicated Cloud SQL for PostgreSQL instance instead of a shared multi-tenant database, trading some operational overhead for full data isolation and independent scaling. Moving connection pooling into Cloud SQL Enterprise Plus eliminated a separate pooling layer entirely, shrinking the security surface and the number of moving parts to maintain. Query Insights turned out to be the highest-leverage feature, letting the team pinpoint exactly which merchant and query were driving cost or latency spikes, while near-zero-downtime instance resizing let them scale per-merchant without disrupting service. The payoff: the database team stopped firefighting plumbing and the business grew 200-300% YoY. #db #cloud #architecture #migration

## [Why Cloud Native Belongs at the Heart of Agentic AI: Lessons from Building a Multi-Agent Security Platform on Kubernetes](https://www.cncf.io/blog/2026/06/17/why-cloud-native-belongs-at-the-heart-of-agentic-ai-lessons-from-building-a-multi-agent-security-platform-on-kubernetes/) 🤓

Treating AI agents as just another Kubernetes workload — rather than inventing bespoke infrastructure — is the core thesis of this build-out of a multi-agent runtime security platform. The stack pairs Falco for syscall-level anomaly detection with Cilium/Hubble for identity-aware network policy, cert-manager and SPIFFE/SPIRE for per-agent mTLS identity, and OPA/Kyverno for policy-as-code governance over what each agent CRD is allowed to do. LangGraph was chosen as the agent runtime after evaluating CrewAI, AutoGen, and LlamaIndex, with OpenTelemetry wired in to trace every policy decision and agent action for audit. It's a concrete blueprint for zero-trust agent governance instead of the usual hand-wavy "secure your agents" advice. #ai #cloud #devops #architecture

## [Getting More from Each Token: How Copilot Improves Context Handling and Model Routing](https://github.blog/ai-and-ml/github-copilot/getting-more-from-each-token-how-copilot-improves-context-handling-and-model-routing/) 🤓

Two architectural changes are cutting Copilot's per-turn cost: prompt caching reuses model state across repeated prefixes instead of recomputing them every request, and tool search lets the model load tool definitions on demand rather than stuffing every schema into context on every turn. The sharper insight is in model routing — switching models mid-conversation breaks the prompt cache, so Copilot's Auto router only switches at natural cache boundaries (first turn, after compaction) and otherwise sticks with one model to keep the cache warm. Rather than binary "easy vs. hard" task labels, the router learns where model capability actually diverges by scoring weaker vs. stronger model outputs across quality dimensions. It's a useful pattern for any team building multi-model agent systems where cache locality and model-switching costs compound fast. #ai #llm #architecture #engineering

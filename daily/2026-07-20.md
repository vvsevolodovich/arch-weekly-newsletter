## [Eclipse Dataspace Components on AWS: Architecture patterns in production](https://aws.amazon.com/blogs/architecture/eclipse-dataspace-components-on-aws-architecture-patterns-in-production/) 👨‍💼

Federated data sharing across organizational boundaries requires stricter isolation than generic microservice patterns provide — the Eclipse Dataspace Components (EDC) connector architecture addresses this by treating each connector instance as an isolated cell: a dedicated ECS/Fargate cluster, Aurora database, Cognito OAuth provider, API Gateway, and VPC deployed as a unit from a single CDK command. The split between control plane and data plane containers is explicit, with encryption enforced at every stage — at rest via Aurora/S3/Secrets Manager, in transit via TLS, and during processing via encrypted environment variables — ensuring data governance guarantees hold regardless of which layer handles a request. Larger enterprises operating in multiple data spaces typically deploy separate connector cells per failure isolation domain or governance boundary, making blast radius and data sovereignty constraints first-class architectural concerns rather than operational afterthoughts. #architecture #cloud #devops #softwareengineering

## [Shipyard: How We Built Slack's Next-Generation EC2 Platform](https://slack.engineering/shipyard-how-we-built-slacks-next-generation-ec2-platform/) 👨‍💼

Configuration drift on long-lived mutable EC2 instances was the root cause of fleet reliability failures at Slack; Shipyard eliminates it by replacing in-place mutation with mandatory replacement — every instance has a bounded lifespan and is automatically rotated, making the fleet always fresh. The architecture is layered: slack-zero is a shared immutable base image (maintained jointly by the Compute Platform, security, and monitoring teams) on top of which service-specific images are built, mirroring the container image model but for bare EC2 — covering AMD64 and ARM Graviton instances across Ubuntu, RHEL, and Amazon Linux. Fleet visibility comes from Peekaboo, an inventory system built on AWS EventBridge, OpenSearch, and Lambda that taps cloud events and instance metadata directly rather than polling Chef Server, providing near-real-time state for a fleet that includes Kubernetes worker nodes and egress stacks that cannot be containerized. #devops #cloud #architecture #engineering

## [Agentic Testing: Where Agents Fit in the E2E Testing Stack](https://slack.engineering/agentic-testing-where-agents-fit-in-the-e2e-testing-stack/) 🤓

The key distinction that determines placement in a CI pipeline: deterministic end-to-end tests enforce journeys, while AI agents verify goals — making them complementary rather than interchangeable. Across 200+ agentic E2E workflows run with Playwright MCP, Playwright CLI, and agent-generated Playwright scripts, Slack found failure rates of 0–12% (Playwright MCP on complex flows), 12–20% (Playwright CLI), and ~8%/~48% (generated scripts on simple/complex workflows), with authentication handling, navigation timing, and session instability the primary CLI failure modes. At $15–30 per run and 10+ minutes per execution, agentic testing is unsuitable for CI regression loops but viable for exploratory coverage where UI churn makes deterministic tests brittle — the architectural recommendation is to keep deterministic tests as the regression gate and reserve agents for goal-oriented validation on critical flows. #softwareengineering #quality #engineering #ai

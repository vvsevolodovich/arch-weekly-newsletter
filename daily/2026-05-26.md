## [Introducing ExtendDB: An Open Source DynamoDB-Compatible Adapter with Pluggable Storage Backends](https://aws.amazon.com/blogs/database/introducing-extenddb-an-open-source-dynamodb-compatible-adapter-with-pluggable-storage-backends/) 👨‍💼

ExtendDB brings the DynamoDB API to environments where the managed service isn't available — developer laptops, air-gapped data centers, disconnected edge sites — with zero application code changes. AWS engineers implemented it in Rust as a layered crate structure: a pure-sync core handles DynamoDB types, validation, and expression evaluation; an engine crate implements the API semantics; and storage backends plug in via a Rust trait, so PostgreSQL ships as the reference backend while community contributors can bolt on Cassandra for horizontal scale without touching the engine. The whole thing compiles to a single binary with no external runtime dependencies, and any existing AWS SDK, CLI, or tooling connects unchanged because ExtendDB speaks DynamoDB's wire protocol at the HTTP level. #db #architecture #cloud #engineering

## [Scaling Airbnb's Identity Graph with a Unified Knowledge Graph Infrastructure](https://medium.com/airbnb-engineering/scaling-airbnbs-identity-graph-with-a-unified-knowledge-graph-infrastructure-ebac467b7836) 🤓

Airbnb's identity graph captures user relationships for Trust and Safety — fraud detection, account linking, data lineage — and the team replaced an externally managed PaaS with an internal JanusGraph cluster backed by DynamoDB for storage and OpenSearch for mixed-type indexing. Hitting latency requirements at scale required two key internals changes to their JanusGraph fork: transactions now exploit DynamoDB conditional writes for lower overhead, and the getMultiSlices interface was rewritten to fetch data in parallel, cutting latency on high-fanout traversals. Load testing of the new service reached 10x the write QPS of the previous solution, and the team shipped distributed tracing directly into the fork to close an observability gap absent from the open-source version. #db #distributed #architecture #engineering

## [Better Experiments with LLM Evals — A Funnel, Not a Fork](https://engineering.atspotify.com/2026/5/better-experiments-with-llm-evals-a-funnel-not-a-fork) 👨‍💼

At Spotify, only 12% of A/B tests produce a shipped positive result and teams roll back 42% of launched experiments before they fully ship, which means evaluation effort spent on experiments that shouldn't run at all is the biggest waste in the pipeline. The fix is positioning LLM evals as a funnel upstream of experiments: automated judges score relevance, coherence, and quality across large candidate sets, catching trust-breaking content or regressions that make a treatment unworthy of a full test, while experiments remain the authority on whether actual user outcomes moved. Calibrating the eval judge against held-out experiment results closes the feedback loop, keeping the judge's signals aligned with real outcome movement rather than drifting on proxy metrics. #ai #softwareengineering #engineering #quality

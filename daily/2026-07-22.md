## [Exploring Hierarchical Interest Representation For Meta Ads Deep Funnel Optimization](https://engineering.fb.com/2026/07/15/ai-research/exploring-hierarchical-interest-representation-for-meta-ads-deep-funnel-optimization/) 🤓

Meta's Hierarchical Interest Representation (HIR) is an upstream embedding layer over the full ads entity graph — users, advertisers, products, services — that learns joint representations connecting latent user interest to the breadth of what advertisers offer in deep funnel ads, plugging directly into the Andromeda, GEM, and Adaptive Ranking Model stack. The technical core is in-house transformer-based graph learning with bias-aware attention and self-supervised cross-view distillation, producing multi-hierarchical interest embeddings across a large-scale graph where LLM-enriched multimodal advertiser content supplements sparse engagement signals to enable generalization to rare and cold-start entities. The bias-aware attention mechanism explicitly corrects for the skew toward popular advertisers inherent in interaction data — a structural fix that most embedding systems leave to post-hoc calibration rather than baking into the learning objective. #ai #architecture #engineering #llm

## [Multi-cloud lakehouse architecture on AWS for Agentic AI, Part 1: Architecture and best practices](https://aws.amazon.com/blogs/big-data/multi-cloud-lakehouse-architecture-on-aws-for-agentic-ai-part-1-architecture-and-best-practices/) 👨‍💼

Agentic AI workloads that need to reason over data across cloud boundaries require a unified metadata catalog as a first-class architectural primitive — the AWS multi-cloud lakehouse approach uses Apache Iceberg as the open table format, AWS Glue as the catalog, and Lake Formation for governance, with AI agents inheriting Lake Formation permissions automatically whether they query through Athena, Redshift Spectrum, or direct Boto3 SDK calls. The newly pre-announced AWS Context service (debuted at the AWS New York Summit 2026) closes the gap between schema discovery and query intent by teaching agents how tables relate, which sources are authoritative, and how users have historically queried them — a semantic layer above the structural catalog. The pattern eliminates the need for parallel data copies or provider-specific query rewrites, letting agents operate through MCP servers against a single logical catalog view spanning multi-cloud storage. #architecture #cloud #db #ai

## [Operating AI/ML Workloads on Kubernetes: A Headlamp Plugin for Kubeflow](https://kubernetes.io/blog/2026/07/13/introducing-headlamp-plugin-for-kubeflow/) 👨‍💼

Kubeflow's native UI surfaces ML pipeline state through its own service layer, hiding the Kubernetes-level Pod conditions and failure reasons that operators actually need during incident response. The Headlamp plugin for Kubeflow reads directly from the Kubernetes API server — surfacing Pod conditions, failure reasons, CPU/memory/GPU resource requests, and volume mount types (PVC, ConfigMap, Secret, emptyDir) — inside a general-purpose Kubernetes UI without an ML intermediary or additional database. The plugin dynamically discovers which Kubeflow API groups are installed on a cluster and renders only those components, making it safe to deploy on partial Kubeflow installations; the pattern is explicitly designed to generalize to any CRD-heavy platform that wants to meet operators in the tool they already use. #devops #cloud #engineering #softwareengineering

## [What's new in Postgres 19](https://planetscale.com/blog/whats-new-in-postgres-19) 🤓

Postgres 19's headline feature is REPACK — a native command that finally solves the long-standing table bloat problem: both VACUUM FULL and CLUSTER previously held an ACCESS EXCLUSIVE lock for the entire table rewrite, blocking all reads and writes for potentially hours on large tables. REPACK CONCURRENTLY rewrites the table under a shared update exclusive lock (the same tier as regular VACUUM), acquiring the exclusive lock only briefly at the final swap — effectively bringing the popular pg_repack extension into Postgres core. Postgres 19 Beta 2 shipped July 16; GA is expected September/October 2026. Caveats: REPACK CONCURRENTLY does not apply to unlogged tables, partitioned tables, or tables without a primary key or replica identity. #db #softwareengineering #architecture

## [Supabase Pipelines is now in Public Alpha](https://supabase.com/blog/supabase-pipelines-public-alpha) 👨‍💼

Supabase Pipelines is a managed change-data-capture product built on Postgres logical replication, powered by Supabase ETL — an open-source CDC framework written in Rust that streams inserts, updates, deletes, and truncates to external analytical systems in near real time. The public alpha launch adds schema change support: Pipelines now detects compatible DDL changes on source tables and applies them to the destination automatically, eliminating the manual schema-sync step that makes most CDC pipelines brittle in production. Google BigQuery is generally available as a destination; ClickHouse, Snowflake, and DuckLake are accessible via early-access request. #db #distributed #architecture #engineering

## [Platform engineering for the agentic enterprise: Managing applications, resources, and AI agents](https://www.cncf.io/blog/2026/07/21/platform-engineering-for-the-agentic-enterprise-managing-applications-resources-and-ai-agents/) 👨‍💼

Internal Developer Platforms built for human developers are structurally unprepared for agentic AI workloads, which introduce non-human consumers requiring GPU/TPU scheduling, model serving infrastructure, MCP gateways, agent guardrails, granular cost attribution, and continuous compliance enforcement — none of which were in the original IDP design surface. The platform engineering response is to extend the IDP to serve AI agents as first-class tenants alongside ML engineers, data scientists, and FinOps practitioners, with Kubernetes providing orchestration, OpenTelemetry observability, SPIFFE workload identity, Dapr resilience, and Falco policy enforcement. The shift from single-persona (developer) to multi-persona (human and agent) platform design is the defining IDP challenge of 2026. #architecture #cloud #devops #ai
